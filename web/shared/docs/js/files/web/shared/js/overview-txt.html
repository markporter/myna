Overview

Summary

Myna is Java web application (.war file) that provides Dynamic web content from both interpreted and compiled JavaScript, as well as complete access to Java libraries.

Goals

To provide an open source rapid web development platform that:

  • Can be distributed as a simple “.war” file.
  • Can leverage Java’s large number of FOSS libraries.
  • Can be easily administered online.
  • Can deployed multiple times on the same hardware (separate instances)
  • Can be deployed to multiple operating systems
  • Can run on JVM 1.3+
  • Can be deployed on to of any Java Servlets container

Myna Specific Features

  • JVM 1.3+ compatibility The Rhino API is accessed directly rather than through Java 6’s scripting API.
  • Minimal setup: Dynamic pages can be created immediately on deployment of the .war file.  There is no need to to set up complicated application definitions.
  • Server-side JavaScript (implemented): Files in the in the application root or sub folders ending with “.sjs” are treated as server-side JavaScript.  JavaScript content is executed server side and sent to there browser.  EX: $res.print(“<html>”); $res.print(“ <body>”); $res.print(“ Today’s date is” + String(new Date()) + “!”); $res.print(“ </body>”); $res.print(“</html>”);
  • Embedded JavaScript (implemented): Files in the in the application root or sub folders ending with “.ejs” are treated as embedded JavaScript.  Embedded JavaScript files can contain any text content (typically HTML) with JSP style escaped dynamic content.
EX:
    <% var today = new Date(); %>
    <html>
        <body>
            Today's date is<%=today%>!"
        </body>
    </html>
  • Embedded JavaScript Blocks (implemented): EJS blocks work like .ejs paged but can be used in JavaScript code to return a string.  All text between <ejs> and </ejs> will be returned as if it was included from a .ejs page.  This makes it much easier to embed other languages like SQL or HTML in JavaScript strings EX: var qry = $lib.sql.query(“test1”,<ejs> select chanid, name, callsign from channel where channum = <%=$req.data.channel_number%> </ejs>);
  • Application lifecycle hooks (implemented): User code can hook into the following events:
  • Request start
  • Request end
  • Session start
  • Application start
  • Database connection pooling (implemented) Future releases will allow the maximum size/age of connection threads to be managed in the administrator.
  • Standard library (partially implemented): Structured library of useful functions.  Client side compatible functions are stored in seperate files so they can be loaded in the browser

EX

$lib.sql.query(datasource,sql,options): Returns a $lib.sql.QueryResult object with metadata and an array of objects representing the result set.  Connection and resultset are closed automatically $lib.string.toJsonString(object): Converts an object to a JSON string.  $lib.string.encryptString (text,password) returns 7-bit ASCII cypher text $lib.file.copyDirectory(srcURI,destURI): Recursively copies the contents of srcURI into destURI

  • Profiling (implemented): A global profiler object is available for profiling code.  An administrator setting can display an execution time summary at the bottom of every page.
  • Script Compilation (implemented): JavaScript files can be compiled and cached to improve performance
  • Web-based Administration (partially implemented): All currently implemented settings can be modified through the provided administrator application
  • Easily extensible: Most of the framework for request handling is implemented in JavaScript, so it is possible to completely re-architect the request lifecycle without re-compiling the Java servlet.
  • Centralized datasources (implemented): Datasources can be defined in the administrator and then referred to by name at runtime
  • Buffered content: Generated content can be retrieved and cleared at any time during page execution, and is sent to the browser at the end of the request in a manner similar to PHP.  This allows cookies and other headers to be sent at any time during the request.
  • Normalized Paths: All file handling us done using String URIs.  This allows easy cross-platform file access.  Also file names that are not absolute are expanded relative to either the application root, or the currently running script
  • Web Archive Deployment (proposed): A Web Archive (.war file) can be created from the current deployed environment with different administrator settings and a subset of the deployed files and datasources
  • Parallel computing features (proposed): Application scope variables are currently implemented in Jakarta’s Java Caching System, so the potential exists for a network distributed caching of application, server and session scoped variables between multiple instance behind a load balancer.

Plans exist for integrating Sequoia (generic database replication) into the datasource manager.

  • Integrated documentation: a NaturalDocs (http://naturaldocs.org) profile will be included that will allow user code documentation to be integrated with the Myna’s static HTML documentation

Rhino (http://www.mozilla.org/rhino/) Specific Features

  • Access any Plain Old Java Object from JavaScript: EX: var file = new java.io.File();
